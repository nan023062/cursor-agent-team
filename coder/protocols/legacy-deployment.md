# 存量项目部署协议

将 Coder 引入已有代码工程（已上线项目、多年积累代码库）的分级部署指南。

**核心原则**：不全量铺开，不停服务，不动存量稳定代码。只对三类模块动手：高频迭代模块、新建模块、即将重构的模块。

---

## 前置条件

- [ ] 团队核心成员了解 Coder 三阶段生命周期（Init / Develop / Evolution）
- [ ] 确定 `.dna/` 文件的维护责任归属（建议：谁改代码谁更新对应 .dna/）
- [ ] 确认 `.dna/` 目录纳入版本管理（git）
- [ ] 确认不与现有团队流程冲突（Coder 是补充，不是替代）

---

## Phase 0: 模块健康审计（Week 1）

使用 `templates/health-audit.md` 对所有模块进行分级。

### 分级标准

| 等级 | 标准 | Coder 策略 |
|------|------|-------------|
| 🟢 **健康** | 边界清晰、低耦合、改起来有信心 | 完整导入 `.dna/` |
| 🟡 **亚健康** | 有些耦合但可控，改的时候小心点 | 只写 `architecture.md` + `pitfalls.md` |
| 🔴 **高危** | 耦合严重、改一处崩三处 | 只写 `danger-map.md`（布雷图） |
| ⚫ **死代码** | 无人碰、可能已废弃 | 标记后不处理 |

### 执行方式

1. 团队核心成员（3-5 人）用半天时间集体评审
2. 逐模块打分，有分歧的按最悲观等级标记
3. 产出：填写完成的 `health-audit.md`

---

## Phase A: 知识抢救（Week 1-2）

目标：把最有价值的隐性知识从人脑转入文件。

### 对 🟢 模块（选 3-5 个高频迭代的）

1. 用 `@coder init` 初始化 `.dna/` 目录
2. 填写 `architecture.md`（描述**当前实际**架构，不是理想架构）
3. 填写 `pitfalls.md`（从团队记忆中挖掘历史教训）
4. 填写 `dependencies.md`（厘清实际依赖关系）
5. 填写 `architecture.md` 的「性能约束」段

### 对 🔴 模块（选最危险的 2-3 个）

1. 创建 `.dna/` 目录，但只放一个文件：`danger-map.md`
2. 由最熟悉该模块的人填写布雷图（见 `templates/dna/danger-map.md`）
3. 不写 architecture.md（架构已漂移到文档无法描述的程度）
4. 不做任何代码修改

### 对 🟡 模块

本阶段不处理。Phase B 验证通过后再逐步覆盖。

---

## Phase B: 新功能试点（Week 3-6）

选择一个**即将开发的新功能**，用完整 Coder Phase 2 流程执行。

### 选择标准

- 是全新模块（不是在旧模块上改）
- 复杂度适中（2-4 周工作量）
- 最好涉及 1-2 个跨程序集接口

### 执行

1. `@coder init` 初始化新程序集
2. 按 Phase 2 主流程开发：上下文加载 → 任务分解 → 架构 → 实现 → 自审查 → 测试 → 记忆
3. 全程记录：流程顺畅度、AI 辅助质量、团队接受度

### 验收

- 代码质量是否优于同期其他模块？
- .dna/ 文件是否被后续开发者（或 AI）实际使用？
- 团队成员是否觉得流程开销合理？

---

## Phase C: 评估与扩展（Week 7+）

### 根据 Phase B 结果决策

| 结果 | 下一步 |
|------|--------|
| 效果明显，团队接受 | 扩展到所有 🟢 模块 + 新建模块 |
| 效果一般，但不排斥 | 仅对新建模块保持，暂不扩展存量 |
| 团队排斥或开销过大 | 仅保留 Phase A 的知识文件，放弃流程层面 |

### 扩展优先级

1. 所有新建模块（零成本，默认使用 Coder）
2. 🟢 高频迭代模块（已有 .dna/，补充 wip.md 即可）
3. 🟡 模块（从 pitfalls + architecture 开始，逐步完善）
4. 🔴 模块走绞杀者迁移（见 `protocols/strangler-migration.md`）

---

## Phase D: 持续治理（每月）

每月执行一次 Phase 3 Evolution，但报告增加存量项目专属指标：

```
## 屎山治理月报 — YYYY-MM
- 🟢 模块数: N（较上月 +/-）
- 🟡 模块数: N（较上月 +/-）
- 🔴 模块数: N（较上月 +/-）
- 🔴→🟢 绞杀者迁移进度: 模块名 旧代码占比 N%
- 本月 pitfalls 新增: N 条
- 本月 danger-map 更新: N 个模块
- 跨模块耦合点变化: +/- N 个
```

---

## 团队适配要点

### .dna/ 文件的维护责任

- **谁改代码谁更新 .dna/**——和"谁写 bug 谁写 pitfall"一个逻辑
- Code Review 时检查：改了代码但没更新 .dna/ → 提醒补充
- 不要求每次都完美更新，但必须保持"不过时"

### 与现有流程的关系

| 现有流程 | Coder 的角色 |
|---------|---------------|
| Code Review | Coder 不替代 CR，自审查门禁是 AI 辅助编码时的附加检查 |
| Jira / 任务管理 | `wip.md` 不替代 Jira，只用于 AI 辅助编码时的会话续接 |
| Git 分支策略 | 如果团队已有 git 规范，以团队规范为准 |
| 编码规范 | 如果团队已有 C# 规范，合并或以团队规范为准 |

### 约束放宽

存量项目以下约束可放宽：

| 原始约束 | 放宽为 |
|---------|--------|
| 先更新架构后写代码 | Hotfix 后 48h 内补 architecture.md |
| 禁止修改程序集以外的代码 | 初期只记录越界，不强制禁止 |
| 拆分阈值触发拆分 | 初期仅在 Evolution 报告中列出，不强制拆分 |
