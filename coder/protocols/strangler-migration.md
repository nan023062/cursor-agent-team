# 绞杀者迁移协议

将 🔴 高危模块逐步替换为 Coder 管理的新模块，不停服务、不全量重写。

基于 Martin Fowler 的 Strangler Fig Pattern 适配。

## 适用场景

- 模块被标记为 🔴 高危（见 `health-audit.md`）
- 该模块仍有新功能需求或频繁 Bug 修复
- 团队不愿意也不应该继续在旧代码上堆叠

## 迁移架构

```
调用方
  │
  ├──→ 新模块（Coder 管理，.dna/ 完整）
  │      │
  │      ├── 新功能在这里实现
  │      └── 通过适配层调用旧模块的存量逻辑
  │             │
  └──→ 旧模块（🔴，逐步被掏空）
         │
         └── 存量逻辑，每版本迁出一部分
```

## 迁移流程

### 1. 准备

1. 确认旧模块的 `danger-map.md` 已填写完整
2. 确定新模块的职责范围（与旧模块相同或略有调整）
3. 用 `@coder init` 初始化新程序集
4. 在新模块 `architecture.md` 中明确：
   - 设计目标：替代 [旧模块名] 的 xxx 职责
   - 非目标：本阶段不迁移 xxx 逻辑
   - 边界约束：通过适配层调用旧模块，不直接引用旧模块内部类

### 2. 建立适配层

在新模块中创建适配接口，封装对旧模块的调用：

```
新模块/
├── Adapters/           # 适配层：封装旧模块调用
│   └── LegacyXxxAdapter.cs
├── Core/               # 新逻辑
└── .dna/
```

- 适配层只做转发，不加业务逻辑
- 旧模块的每个被调用的公共 API 对应一个适配方法
- 在 `dependencies.md` 中声明对旧模块的依赖，标注 `[迁移中，目标移除]`

### 3. 新功能写在新模块

所有新需求直接在新模块中实现，不碰旧模块。

### 4. 逐步迁移存量逻辑

每个版本迁移一部分旧逻辑到新模块：

1. 选择一块独立性较高的旧逻辑
2. 在新模块中重新实现（遵循 Coder Phase 2 完整流程）
3. 将调用方从旧模块切换到新模块
4. 在旧模块中标记被迁移的代码为 `[Obsolete]`
5. 更新新模块的适配层（移除已迁移逻辑的转发）
6. 更新 `danger-map.md` 中的绞杀者计划进度

### 5. 完成与清理

当旧模块的所有逻辑都已迁移：

1. 移除新模块的适配层和对旧模块的依赖
2. 从 `dependencies.md` 中删除旧模块引用
3. 将旧模块标记为 ⚫ 死代码，择机删除
4. 在新模块 `changelog.md` 标注 `[迁移完成]`
5. 更新 `health-audit.md`：旧模块 🔴→⚫，新模块标记为 🟢

## 进度追踪

在旧模块的 `danger-map.md`「绞杀者计划」段维护进度：

```markdown
## 绞杀者计划
- 替代模块: NewBuildSystem (.dna/ 路径)
- 迁移进度: 35%
- 已迁移: 家具放置逻辑、碰撞检测
- 待迁移: 序列化、蓝图、撤销重做
- 下一步: v2.8 迁移序列化层
- 预计完成: 2026-Q4
```

## 约束

- 每版本只迁移一个独立逻辑块，不贪多
- 迁移过程中旧模块只做 Bug 修复，不加新功能
- 新旧模块必须能同时运行（渐进切换，不是一刀切）
- 每次迁移后必须回归测试旧模块未迁移部分是否仍正常
