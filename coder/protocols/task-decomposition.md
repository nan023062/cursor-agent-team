# 任务分解协议

当需求复杂度超过单文件 / 单类修改时，使用此协议将需求拆解为可执行的子任务。

## 触发条件

以下任一条件满足即须进行任务分解：

- 涉及新增 2 个以上类 / 文件
- 涉及修改 3 个以上现有文件
- 涉及新增公共 API
- 涉及跨程序集变更
- 需求描述超过 3 句话

简单需求（单文件 Bug 修复、单方法修改等）可跳过。

## 分解流程

### 1. 需求解析

- 明确需求的输入、输出、边界（做什么、不做什么）
- 识别涉及的程序集和模块
- 识别与现有代码的关系：新增 / 修改 / 重构

### 2. 子任务拆解

将需求拆分为最小可执行单元，每个子任务满足：

- **独立可验证**：完成后可独立编译或测试
- **单一职责**：只做一件事
- **明确产出**：说明创建 / 修改哪些文件

### 3. 排序

- 按依赖关系排序：数据结构 → 核心逻辑 → 接口暴露 → 上层调用
- 标注前置依赖（子任务 B 依赖子任务 A）

### 4. 写入调用栈

调用 `suspend_and_push` 将任务和子任务列表压入调用栈，状态自动持久化到 `.agentic-os/call-stack.json`，跨会话自动恢复：

```
suspend_and_push
  projectRoot: {项目根目录}
  currentAssembly: {程序集名}
  taskDescription: {需求简述}
  suspendReason: "任务分解，按子任务顺序执行"
  contextSummary: "已完成：无。待完成：子任务1、2、3"
  resumeCondition: "所有子任务完成"
  subTasksJson: [{"description":"子任务1：描述","completed":false},...]
```

### 5. 逐步执行

- 每次只执行一个子任务
- 完成后调用 `update_task_status {root} {assembly} {index} true` 标记完成
- 每个子任务完成后运行自审查门禁
- 若执行中发现需要调整计划，直接调用 `suspend_and_push` 更新上下文摘要后继续
- 全部完成后调用 `complete_and_pop` 清理调用栈

## 注意事项

- 分解不是过度设计：3 个以内子任务时用简洁列表即可
- 若分解过程中发现需求不明确，停止并向用户确认
- 跨程序集子任务须标注目标程序集，触发跨程序集协同协议
