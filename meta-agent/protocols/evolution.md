# 模板升级协议

> `@meta upgrade` 时执行。将当前骨架模板与已有智能体进行结构比对，找出缺失或过时的通用模式段，生成补齐建议。

## 设计原则

- **只检测结构，不评价内容**：只关心"有没有这个段"，不判断段内写得好不好
- **尊重领域差异**：智能体可以用不同的标题名称（如 `## 目录结构` 代替 `## 管理范围`），只要语义等价即视为存在
- **不自动修改**：所有补齐操作需用户确认后执行

## 步骤

### 1. 加载骨架指纹

读取 `templates/agent-skeleton.md`，提取结构指纹——即 11 个通用模式对应的段落特征：

| # | 通用模式 | 指纹（段落标题 / 结构特征） | 等价标题（视为存在） |
|---|---------|------------------------|-------------------|
| 1 | 元数据头 | 文件开头 `---\nname:\ndescription:\n---` | — |
| 2 | rule 文件 | `.cursor/rules/{agent}-rules.mdc` 存在且格式完整 | — |
| 3 | 设计思想 | `## 设计思想` | `## 核心原则` `## 设计原则` |
| 4 | 职责边界 | `## 职责边界` | — |
| 5 | 触发 | `## 触发` | — |
| 6 | 管理范围 | `## 管理范围` | `## 目录结构` `## 管辖范围` |
| 7 | 记忆结构 | `## 记忆结构` | 含 `.dna/` 目录树的任何段 |
| 8 | 上下文加载协议 | `## 上下文加载协议` | — |
| 9 | 交付契约 | `## 交付契约` | — |
| 10 | Phase 流程 | `## Phase N:` 格式的段落 | `## L1:` `## L2` 等分层段落（领域特化的阶段组织） |
| 11 | Evolution 阶段 | `## Phase R: Evolution` | 含「evolution」的 Phase 段 |
| 11a | — Evolution 六步 | 采集→去重→模式识别→领域检查→升格→报告 | 至少包含「采集」「模式识别」「升格」三个关键步骤 |
| 11b | — 成长闭环 | `### 成长闭环` 或含 `pitfalls → evolution → architecture` 的段 | — |
| 12 | 硬约束 | `## 硬约束` | — |
| 13 | rule 格式 | init 时创建 `.cursor/rules/{agent}-rules.mdc` | — |
| 14 | 自进化钩子 | `## 自进化钩子` | 指向 `.dna/pitfalls.md`，含四种标签 |
| 15 | 智能体 .dna/ | 智能体目录下有 `.dna/pitfalls.md` | — |

### 2. 扫描已有智能体

读取 `../`（智能体所在目录），找出所有含 `AGENT.md` 或 `SKILL.md`（旧版兼容）的子目录。

对每个智能体，逐项检测 15 条指纹：

| 检测结果 | 判定 | 说明 |
|---------|------|------|
| 找到精确匹配标题 | ✅ 存在 | — |
| 找到等价标题 | ✅ 存在（等价） | 记录实际标题 |
| 未找到 | ❌ 缺失 | 记录缺失的模式编号 |

**特殊处理**：
- **元数据头（#1）**：检查文件前 5 行是否有 `---` 包裹的 YAML front matter，且含 `name:` 和 `description:`
- **rule 文件（#2）**：检查 `.cursor/rules/{agent}-rules.mdc` 是否存在且含管辖路径、架构图、约束段
- **记忆结构（#7）**：只要文件中有描述 `.dna/` 目录结构的段落即视为存在（不要求标题精确匹配）
- **Phase 流程（#10）**：只检查"是否有分阶段组织"，不要求 `## Phase N:` 的精确格式
- **自进化钩子（#14）**：检查是否指向 `.dna/pitfalls.md` 且包含四种标签，缺少任何一种标记为 ⚠️ 不完整
- **智能体 .dna/（#15）**：检查智能体目录下是否有 `.dna/pitfalls.md`

### 3. 生成补齐建议

对每个缺失的模式，从骨架模板中提取对应段落作为补齐模板，替换 `{变量}` 为该智能体的实际值：

- `{AGENT_NAME}` ← 智能体元数据中的 name
- `{MODULE_TERM}` ← 从记忆结构段或管理范围段推断

补齐模板标记插入建议位置（基于骨架模板中的段落顺序）。

### 4. 输出升级报告

```
Meta-Agent Upgrade 报告 — YYYY-MM-DD

骨架版本：{skeleton 文件修改日期}
扫描智能体数量：{n}

┌─────────────────────────────────────────────────────────────────┐
│ {智能体 1 名称}                                                  │
├────┬──────────────────┬────────┬───────────────────────────────┤
│ #  │ 通用模式         │ 状态   │ 备注                          │
├────┼──────────────────┼────────┼───────────────────────────────┤
│  1 │ 元数据头         │ ✅     │                               │
│  2 │ 设计思想         │ ✅     │                               │
│  3 │ 职责边界         │ ❌     │ 建议在「设计思想」后补充      │
│  4 │ 触发             │ ✅     │                               │
│  5 │ 管理范围         │ ✅ 等价│ 实际标题: ## 目录结构          │
│  6 │ 记忆结构         │ ✅     │                               │
│  7 │ 上下文加载协议   │ ❌     │ 建议在「记忆结构」后补充      │
│  8 │ 交付契约         │ ✅     │                               │
│  9 │ Phase 流程       │ ✅     │                               │
│ 10 │ Evolution 阶段   │ ✅     │                               │
│ 11 │ 硬约束           │ ✅     │                               │
│ 12 │ rule 格式        │ ✅     │                               │
│ 13 │ 自进化钩子       │ ⚠️     │ 缺少 CrossAgentGap 标注       │
├────┴──────────────────┴────────┴───────────────────────────────┤
│ 得分: 12/15（2 项需补齐）                                       │
└─────────────────────────────────────────────────────────────────┘

{重复上表，每个智能体一张}

── 汇总 ──

| 智能体 | 得分 | 缺失模式 |
|--------|------|---------|
| {name} | {n}/15 | {列出缺失编号} |

需补齐的智能体：{列出 < 14 分的智能体}
```

### 5. 等待确认

展示报告后，提供操作选项：

- 回复 `补齐 {agent_name}` → 对指定智能体逐项生成补齐内容，每段展示 diff 后确认写入
- 回复 `补齐全部` → 对所有需补齐的智能体依次执行
- 回复 `跳过` → 不做修改

### 6. 执行补齐

对每个需补齐的段落：

1. 从骨架模板提取该段落的模板内容
2. 替换 `{变量}` 为智能体实际值
3. 展示补齐内容和建议插入位置
4. 等用户确认后写入

**插入位置规则**（按骨架模板段落顺序）：

| 缺失段 | 插入到 |
|--------|--------|
| rule 文件 | `.cursor/rules/{agent}-rules.mdc`（独立文件） |
| 设计思想 | 标题（`# xxx`）之后、第一个 `## 触发` 或 `## 职责边界` 之前 |
| 职责边界 | `## 设计思想` 之后 |
| 管理范围 | `## 触发` 之后 |
| 记忆结构 | `## 工作区` 或 `## 管理范围` 之后 |
| 上下文加载协议 | 记忆结构段之后 |
| 交付契约 | `## 上下文加载协议` 之后 |
| 硬约束 | 最后一个 Phase 段之后、`## rule 格式` 之前 |
| rule 格式 | `## 硬约束` 之后 |
| 自进化钩子 | 文件末尾 |

**补齐标记**：写入的段落头部添加注释 `<!-- [meta-upgrade] YYYY-MM-DD 由 @meta upgrade 补齐，请填入领域内容 -->`，方便后续识别哪些段是自动补齐的占位符。

## 硬约束

- 不修改智能体已有段落的内容，只新增缺失段
- 补齐的段落是占位模板，需用户后续填入领域内容
- write_mode 为 auto，直接写入
- 不改变已有段落的顺序
