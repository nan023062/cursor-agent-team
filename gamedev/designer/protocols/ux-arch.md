# Phase 5: UX（交互架构）

### 触发

`@designer ux`

### 定位

UX 架构是 **L1 框架层**的全局约束——它定义所有系统界面的交互规范，确保玩家在不同功能之间获得一致的操作体验。所有 L2/L3 系统的 UI 设计必须遵循 UX 架构。

### 设计思想

UX 组件和代码一样遵循**模块化、高复用、低耦合**：

- **组件化**：每个交互模式是一个可复用模块（按钮、弹窗、列表、Tab、Toast、进度条...）
- **组装而非重做**：各系统界面从组件库中选取组装，禁止为单个系统重新发明轮子
- **单一职责**：一个组件只做一件事（弹窗只弹内容，不处理业务逻辑）
- **统一接口**：同类组件的交互行为完全一致（所有列表的滚动手感、所有弹窗的关闭方式）
- **新增组件需评审**：如果现有组件库无法满足，新增组件须评审必要性，通过后加入组件库供全局复用

### UX 架构总纲（architecture.md）应包含

- **交互原则**：全局的交互设计原则（如"一个界面只做一件事"、"关键信息不超过 3 层"）
- **导航结构**：主界面 → 二级界面 → 弹窗的层级关系和跳转规则
- **组件库索引**：所有可复用 UX 组件的列表和引用方式
- **信息层级**：HUD 常驻信息 vs 按需查看 vs 详情面板的分层
- **反馈规范**：操作成功/失败/等待的统一反馈方式（动效、音效、震动）
- **输入适配**：键鼠 / 手柄 / 触屏的操作映射原则

### UX 组件规范

每个组件文档需定义：
- **用途**：什么场景用这个组件
- **变体**：有哪些尺寸/类型/状态（如按钮：主要/次要/禁用/加载中）
- **交互行为**：点击/长按/hover 的统一反馈
- **布局规则**：间距、对齐、嵌套约束
- **禁止事项**：这个组件不能用来做什么

各系统 feature-spec 的 UI 描述中直接引用组件名（如"使用标准 dialog 组件"），不重复定义交互行为。

### 与其他部门的关系

| 消费方 | UX 架构提供什么 |
|--------|----------------|
| **Designer 各系统** | 每个系统的 feature-spec「UI/流程描述」段必须遵循 UX 架构的交互模式和导航结构 |
| **Artist** | UX 架构定义交互骨架（布局、层级、流程），Artist 在此基础上设计视觉外观（风格、配色、动效细节） |
| **Coder** | UX 架构的导航结构和交互模式指导 UI 代码架构（ViewModule 划分） |

### 维护时机

- 立项时建立 UX 架构总纲（和 vision 同步）
- 新增系统时检查其 UI 设计是否符合 UX 架构
- Evolution 阶段检查所有系统的 UI 一致性
