# Code Review 协议

AI 生成的代码在合并前须经过 Review。本协议定义 Review 的触发条件、检查维度和处置流程。

**核心原则**：不检查大模型已经擅长的东西（命名、格式、语言惯用法），只检查它容易出错的东西（边界、业务正确性、隐式耦合、性能实际表现）。

---

## 触发条件

| 场景 | Review 级别 | 说明 |
|------|------------|------|
| AI 实现简单 Bug 修复（单文件、逻辑明确） | **免 Review** | 边界守卫已覆盖 |
| AI 实现常规功能（单程序集内） | **自 Review** | AI 完成后按下方 Checklist 自查，输出 Review 摘要 |
| AI 实现涉及公共 API 变更 | **人工 Review** | 接口设计影响所有调用方，必须人审 |
| AI 实现跨程序集变更 | **人工 Review** | 集成点容易出隐式耦合 |
| AI 实现核心算法 / 性能敏感逻辑 | **人工 Review** | AI 无法真机验证性能 |
| `wip.md` 中标注 `[需 review]` 的子任务 | **人工 Review** | 开发时已预判需要人审 |

---

## Review Checklist

### AI 自 Review（每次实现后、提交前执行）

聚焦 AI 容易遗漏的维度：

- [ ] **业务意图**：实现是否匹配需求描述的意图，而非仅满足字面表述
- [ ] **边界完整性**：新增代码是否全部在程序集目录内，无越界引用
- [ ] **接口契约**：新增/修改的公共 API 参数、返回值、异常行为是否与 `architecture.md` Public API 段一致
- [ ] **隐式依赖**：是否引入了 `dependencies.md` 未声明的隐式依赖（反射、字符串路径、全局静态访问）
- [ ] **空值安全**：公共 API 入口是否有参数校验，返回值是否可能为 null 而调用方未处理
- [ ] **并发安全**：若涉及多线程 / 异步，共享状态是否有竞态风险
- [ ] **资源泄漏**：新创建的对象（GameObject、集合、流）是否有对应的释放路径

自 Review 通过后，在 `wip.md` 或 commit message 中标注：`[self-reviewed]`

### 人工 Review（由维护者或指定 Reviewer 执行）

在 AI 自 Review 基础上，增加人工才能判断的维度：

- [ ] **设计合理性**：方案是否是解决问题的合理方式，还是 AI 在"绕路"
- [ ] **业务正确性**：游戏逻辑是否正确（数值、状态转换、交互规则）——AI 不知道"游戏应该怎么玩"
- [ ] **性能实感**：关键路径的性能是否需要真机验证（AI 只能估算，不能 Profile）
- [ ] **存量兼容**：对于已上线项目，变更是否影响现有存档 / 配置 / 网络协议的兼容性
- [ ] **可读性**：代码是否让人（不只是 AI）能快速理解意图

## Review 结果处置

| 结果 | 处置 |
|------|------|
| ✅ 通过 | 合并，正常写 changelog |
| ⚠️ 有建议但不阻塞 | 合并，建议项记入 `wip.md` 作为后续优化任务 |
| ❌ 不通过 | 打回，在 `wip.md` 标注问题，AI 根据 Review 反馈修改后重新提交 |
| ❌ 发现架构问题 | 打回，先更新 `architecture.md`，再重新实现 |

### Review 发现的教训

- 若 Review 中发现的问题具有普遍性（不是一次性的笔误），记入 `pitfalls.md`
- 若同类问题在 3 次以上 Review 中重复出现，在 Phase 3 Evolution 时建议升级为 `architecture.md` 的边界约束

## 单人 + AI 场景的简化

单人开发时没有"别人"来做人工 Review，采用以下替代策略：

1. **AI 自 Review** 每次必做（7 项 checklist）
2. **关键变更自审**：涉及公共 API 或性能敏感逻辑时，要求 AI 输出 Review 摘要，开发者快速扫读
3. **延迟批量 Review**：每周花 30 分钟扫读本周所有 AI 生成的 `[self-reviewed]` 提交，抽查 2-3 个
4. **Phase 3 兜底**：Evolution 阶段的架构漂移检测能捕获自 Review 遗漏的问题
