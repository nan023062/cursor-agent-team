# 跨程序集协同协议

当一个需求涉及多个程序集的变更时，使用此协议替代单程序集 Phase 2 流程。

## 前置条件

在 Phase 2 步骤 2（分解任务）中识别出跨程序集需求。

## 协同流程

### 1. 影响分析

- 列出涉及的所有程序集
- 为每个程序集标注变更类型：

| 变更类型 | 说明 |
|---------|------|
| 新增 API | 在被依赖方新增公共接口 |
| 修改 API | 修改现有公共接口（可能 Breaking） |
| 仅调用 | 只调用其他程序集的现有 API |
| 无变更 | 不涉及代码修改，仅更新文档 |

- 绘制依赖方向：A → B 表示 A 依赖 B

### 2. 排序执行

按依赖关系从底层到上层排序：

- 被依赖方优先（如 `vena.core` 先于 `vena.module.*`）
- 同层无依赖关系的程序集可顺序执行
- 修改 API 的程序集必须先于仅调用的程序集

### 3. 逐集执行

每个程序集独立执行 Phase 2 主流程：

1. 切换到该程序集，加载其上下文（重新执行上下文加载协议）
2. 只修改该程序集目录内的代码
3. 更新该程序集的 `architecture.md` 和 `dependencies.md`
4. 完成后验证接口兼容性

**关键约束**：每次只聚焦一个程序集的上下文，不同时操作多个程序集。

### 4. 集成验证

全部程序集执行完毕后：

- 检查所有新增/修改的跨程序集接口是否匹配（参数类型、返回值、命名空间）
- 运行涉及的所有程序集的测试
- 检查无循环依赖引入（A→B→C→A）
- 若下游程序集需要上游尚未完成的 API，必须先在上游完成后再继续

### 5. 记录

- 每个程序集的 `changelog.md` 标注 `[跨程序集]` 并列出关联程序集名
- 若有 Breaking Change，在所有受影响程序集的 `changelog.md` 标注 `[BREAKING]`
- 在 `wip.md` 中记录整体进度，标注各程序集完成状态

## 并行模式（团队场景）

当团队有多名开发者时，互不依赖的程序集可由不同人同时执行：

1. 在步骤 1（影响分析）中，将程序集按依赖关系划分为**无依赖的独立组**
2. 每组分配给不同开发者（各自带自己的 AI Agent）
3. 组内仍按串行执行（依赖方向排序）
4. 全部组完成后，统一执行步骤 4（集成验证）

**约束**：有依赖关系的程序集不可并行——下游必须等上游完成。

示例：需求涉及 A → B、C → B、D（独立）

```
开发者 1: B（底层）→ A（上层）
开发者 2: D（独立，可与开发者 1 并行）
开发者 1 完成 B 后: 开发者 2 可接手 C（依赖 B）
```

单人场景下此节无效，退化为默认的串行执行。

---

## 异常处理

| 场景 | 处理方式 |
|------|---------|
| 执行中发现需要额外程序集 | 停止，重新执行步骤 1（影响分析） |
| 上游 API 设计需要调整 | 回退上游程序集，重新设计后从上游重新执行 |
| 出现循环依赖 | 停止，提取公共接口到更底层程序集或引入事件解耦 |
