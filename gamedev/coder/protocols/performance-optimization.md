# 性能优化协议

当性能问题被发现后（来自 Profiler、QA 报告、线上监控或用户反馈），程序侧按本协议执行优化。

**职责边界**：
- 跑 Profiler、抓取性能数据、基准测试、验证优化效果 → **Tester**
- 分析瓶颈、设计方案、实现优化代码 → **Coder（本协议）**

---

## 触发条件

以下任一情况触发本协议：

- 用户/QA 报告帧率不达标或卡顿
- Profiler 数据显示某方法超出 `architecture.md` 性能预算
- Phase 3 Evolution 中趋势度量发现性能指标恶化
- 线上监控告警（崩溃率、ANR 率上升）

## 优化流程

### 1. 定位（Tester 提供数据，Coder 接手分析）

**Tester 负责**：运行 Profiler，截取性能数据，识别瓶颈方法/模块。

**Tester 交给 Coder 的输入**（写入 `wip.md`）：

```markdown
## 性能优化：[问题简述]
- 开始: YYYY-MM-DD
- 状态: 进行中
- 瓶颈定位:
  - 方法: ClassName.MethodName()
  - 当前耗时: Xms（预算: Yms）
  - 触发场景: 描述（如"房间内 200+ 家具时"）
  - Profiler 关键发现: 描述（如"GC Alloc 8KB/帧，主要来自 List 扩容"）
- 子任务:
  - [ ] 分析代码路径，找出热点
  - [ ] 设计优化方案
  - [ ] 实现优化
  - [ ] 验证效果
```

### 2. 分析（AI 执行）

AI 读取瓶颈方法的代码，按以下维度分析：

| 维度 | 检查项 |
|------|--------|
| **GC 分配** | 热路径中是否有 new、LINQ、Lambda、字符串拼接、装箱 |
| **算法复杂度** | 是否存在 O(n²) 或更差的循环嵌套 |
| **冗余计算** | 是否有可缓存的重复计算 |
| **调用频率** | 是否每帧调用但实际可降频（隔帧/脏标记） |
| **数据结构** | 集合类型是否匹配访问模式（List vs Dictionary vs Array） |
| **Unity 陷阱** | GetComponent 缓存、Camera.main 缓存、物理查询复用 |

分析结果输出为方案，包含：预期收益、实现成本、风险。

### 3. 实现（AI 执行，走 Phase 2 主流程）

按 Phase 2 必做流程实现优化代码。额外约束：

- 优化前后的代码行为必须一致（不改逻辑，只改性能）
- 若优化需要改变接口或数据结构，先更新 `architecture.md`
- 大型优化（如从 OOP 改 ECS、引入 Job System）视为重构，走 Refactor 流程

### 4. 验证（Tester 负责）

Tester 再次运行 Profiler，对比优化前后：

- 目标方法耗时是否降到预算内
- 是否引入新的性能问题（拆东墙补西墙）
- 在目标设备（低端机）上验证
- 验证结果反馈给 Coder，未达标则回到步骤 2 重新分析

### 5. 记录

无论优化成功还是失败，都写入 `pitfalls.md`：

```markdown
- [YYYY-MM-DD] #performance 优化简述
  - 根因: 瓶颈原因
  - 修复: 优化手段（如"List 预分配 + 缓存 GetComponent"）
  - 效果: Xms → Yms（目标设备）
  - 影响: 涉及的文件/方法
```

同时更新 `architecture.md` 性能约束段（如果预算需要调整）。

---

## 常见优化手段速查

供 AI 在步骤 2 分析时参考，按收益/成本比排序：

| 优先级 | 手段 | 典型收益 | 成本 |
|--------|------|---------|------|
| 1 | 缓存（GetComponent、Camera.main、计算结果） | 高 | 极低 |
| 2 | 消除 GC（预分配集合、对象池、避免装箱） | 高 | 低 |
| 3 | 降频（脏标记、隔帧更新、LOD 逻辑） | 高 | 低 |
| 4 | 数据结构替换（List→Array、Dictionary→NativeHashMap） | 中 | 中 |
| 5 | 算法优化（空间哈希、BVH、分帧处理） | 高 | 中 |
| 6 | 多线程（Job System、Burst Compiler） | 极高 | 高 |
| 7 | 架构级重构（OOP→ECS/DOD） | 极高 | 极高 |

**原则：先摘低垂果实（1-3），再考虑重型优化（4-7）。**

---

## 与其他协议的关系

- 优化实现走 **Phase 2 主流程**（边界守卫、Code Review 照常）
- 架构级优化走 **Refactor 流程**
- 优化教训进入 **pitfalls.md**，Phase 3 Evolution 时识别高频性能模式
- 性能预算声明在 **architecture.md** 性能约束段
