# Agentic OS — 设计思路

## 一句话定义

**Agentic OS 是面向 AI Agent 的工作区操作系统。** 它管理的不是代码，而是工作区——任何类型的内容单元（代码、美术、策划、音视频……）都是模块，`.dna/` 是模块的身份编码，调用栈是 AI 的工作记忆。

---

## 核心问题

当前 AI Agent 协作面临三个根本性问题：

1. **上下文断裂**：对话窗口关了，AI 忘记自己在做什么、做到哪了
2. **边界失控**：AI 能看到一切，也可能改错一切，没有物理级的访问隔离
3. **规则不跟人走**：规则在 prompt 里，换个 AI / 换个会话，规则就丢了

Agentic OS 用三层确定性程序解决这三个问题。

---

## 三层架构

```
┌─────────────────────────────────────────┐
│              Agentic OS                  │
│                                          │
│  ① 上下文管控（静态）                     │
│     读 .dna/ → 按拓扑和视界过滤 → 喂给 AI  │
│     AI 不该看的，物理上就看不到             │
│                                          │
│  ② 任务栈调度（动态）                     │
│     push / pop / suspend / resume        │
│     跨会话持久化，任何 AI 接手都能续接      │
│                                          │
│  ③ 拓扑感知（结构）                       │
│     模块间的依赖关系 → 执行顺序 → 视界等级  │
│     循环依赖检测、执行计划生成              │
│                                          │
└─────────────────────────────────────────┘
         ↑                    ↑
    MCP 接口              文件系统
  (任何 AI 可调用)     (.dna/ + call-stack.json)
```

---

## 核心概念：模块

**模块（Module）= 任何可独立操作的工作内容单元。**

Agentic OS 不关心模块里装的是什么——它只关心模块有没有 `.dna/`，以及模块之间的依赖关系。

| 行业 | 模块示例 |
|------|----------|
| 游戏开发 | `Code/Core`、`Art/Characters`、`Design/LevelConfig`、`Doc/GDD` |
| 影视工作室 | `Script/Act1`、`VFX/Explosion`、`Sound/BGM`、`Edit/Trailer` |
| 公众编辑部 | `Article/TechReview`、`Design/Cover`、`Review/FactCheck` |
| 小程序开发 | `Frontend/Login`、`Backend/API`、`Config/Env`、`Doc/API` |

---

## 核心概念：.dna/

### 类比理解

**人体类比**：每个模块是一个器官，`.dna/` 是它的神经元——不是器官本身，而是器官对外沟通的接口。不同器官的神经元传递不同信号（视觉/触觉/运动），不同模块的 `.dna/` 有不同的职责声明形态。

**企业类比**：每个模块是一个岗位，`.dna/` 是它的员工守则和工作手册——新员工入职第一天读工作手册，员工离职了工作手册留给继任者。

### 设计本质

> `.dna/` 不是模块的"内容"，而是模块的**身份编码**。

- `architecture.md`（L3 永久规则）= 基因表达 = 岗位职责说明书
- `pitfalls.md`（L1 原始经验）= 免疫记忆 = 错误案例手册
- `dependencies.md` = 神经突触连接 = 协作部门清单
- `changelog.md` = 生长记录 = 变更日志
- `wip.md` = 当前活跃的信号 = 工作交接文档

### 规则跟着工作区走

**关键设计决策：规则绑定在模块上，不绑定在 AI 上。**

```
传统方式：规则在 Agent 的 prompt 里 → 换 AI 就丢了
Agentic OS：规则在模块的 .dna/ 里 → 换 AI 无所谓，规则还在
```

无论哪个 AI 来操作这个工作区，都必须先读 `.dna/`。模块来来去去，内容千变万化，但 `.dna/` 是恒定的身份。

---

## 核心概念：视界分级

视界分级是 Agentic OS 最关键的安全机制——**物理级的上下文隔离**。

| 访问级别 | 触发条件 | 可见内容 | 类比 |
|----------|----------|----------|------|
| **Current** | 当前操作的模块 | 全部 `.dna/` + 全部内容文件 | 自己部门，完全权限 |
| **SharedOrSoft** | Shared/Soft 边界邻居 | 全部 `.dna/` + 职责声明相关文件 | 兄弟部门，可看对方文档 |
| **HardDependency** | Hard 边界依赖 | 仅 `architecture.md` 职责声明段 | 外部供应商，只看接口文档 |
| **Unlinked** | 无依赖关系 | 物理拦截，什么都看不到 | 不相关的公司，无权访问 |

**职责声明段**因模块类型而异，OS 自动识别：

- 代码模块：`## Public API`
- 策划模块：`## 交付契约`
- 美术模块：`## 资产规范`
- 自定义：`## 职责声明`

类比人体的**血脑屏障**——不是所有信号都能进入大脑，硬边界模块的内部实现在 MCP 服务端就被物理屏蔽。

---

## 核心概念：调用栈

调用栈解决的是 AI 的**跨会话认知连续性**问题。

```
栈帧 #0（挂起）: 制作宣传片
  挂起原因: 等待 VFX/Explosion 完成爆炸特效

栈帧 #1（当前）: VFX/Explosion
  子任务: ✓ 粒子设计 / ○ 渲染测试 / ○ 交付审核
```

- 调用栈存储在文件系统（`{projectRoot}/.agentic-os/call-stack.json`）
- 不依赖任何平台，任何 AI、任何 IDE、任何会话都能恢复
- AI 关闭对话后重新打开，`get_call_stack` 立刻找回上次进度

## 与 n8n / Coze 的本质区别

| | n8n / Coze | Agentic OS |
|--|------------|------------|
| **调度主体** | 平台/引擎 | AI 自己 |
| **流程定义** | 人预先画好的图 | AI 运行时动态决策 |
| **状态保存** | 平台托管 | 写入项目文件系统 |
| **执行模型** | 自动化流水线 | 认知调用栈 |
| **适用场景** | 已知的、可重复的业务流程 | 未知的、探索性的复杂任务 |

**n8n/Coze 是给 AI 画好路走，Agentic OS 是给 AI 一个大脑让它自己找路，并且记住走到哪了。**

## 递归层级 Agent 模型

每个模块可以被看作一个 Agent 的操作区。Agentic OS 天然支持递归层级结构：

```
项目根节点
├── 程序（大类模块）
│   ├── 架构
│   ├── Gameplay
│   │   ├── 背包系统
│   │   ├── 战斗系统
│   │   └── 任务系统
│   ├── UI 系统
│   └── 渲染
├── 美术（大类模块）
│   ├── 角色
│   ├── 场景
│   └── UI 美术
└── 策划（大类模块）
    ├── 数值
    └── 关卡
```

大类模块是"调度 Agent 协作的"，内部模块是"调度 Agent 内部任务分工的"。这映射了人类真实的组织架构——大职责分类下有二级分类，大型项目可以拆得更细。

调用栈在层级中的行为：

```
栈帧 #0  项目: "完成 1.0 版本"
  挂起原因: 等待程序完成核心框架

栈帧 #1  程序: "搭建核心框架"
  挂起原因: 等待架构模块完成注册系统

栈帧 #2  架构: "实现模块注册系统"  ← 当前执行
```

## `.dna/` 需要人工设置

这是一个有意的设计决策，不是缺陷。

### 为什么要人工

> **架构决策不应该由 AI 自动推断，因为它是主观的、有意图的。**

两个模块之间"应不应该有依赖"，是架构师的判断，不是扫描文件就能得出的结论。人工设置 `.dna/` 本质上是在**把架构意图显式化**，让 AI 能够遵守而不是猜测。

### 权衡取舍

```
全自动推断 DNA          手工维护 DNA
（AI 猜测结构）    ←→   （人显式声明意图）
  低摩擦                  高可信
  易出错                  准确反映架构意图
  适合探索期              适合有明确架构的项目
```

当前选择偏右侧——用少量人工投入换取 AI 行为的高可预测性。

### 降低摩擦的演进方向

| 方向 | 描述 |
|------|------|
| AI 辅助初始化 | `register_module` 时 AI 根据目录内容自动草拟 DNA，人审核确认 |
| 约定优于配置 | 目录结构本身隐含 DNA（`art/characters/` 自动推断为美术-角色模块） |
| 增量维护 | DNA 不需要一次性完整，AI 在工作过程中逐步补充完善 |
| 模块类型模板 | 不同类型模块有预设 DNA 模板，降低初始成本 |

## Agent 可替换性

Agentic OS 最重要的设计特性：**Agent 可替换，规则不丢。**

```
场景：Cursor Agent 做了一半背包系统，挂起了
      换成 Claude Code 来续接

Agentic OS：
  get_call_stack → 这是上次的进度
  read_dna       → 这是模块的规则和教训
  get_module_context → 这是你能看到的上下文

Claude Code 无缝续接。
```

一切状态在文件系统里，不在任何 Agent 的记忆里。这是 Agentic OS 与所有"Agent 框架"（Swarm/AutoGen/CrewAI）的根本区别——那些框架的协作是无记忆的、扁平的、单次会话的。Agentic OS 是有记忆的、有层级的、有约束的、跨会话持久的。

---

## 总结

```
有记忆（.dna/ + call-stack.json）
  × 有层级（递归模块树）
  × 有约束（拓扑依赖 + 视界过滤）
  × 跨会话（文件系统持久化）
  × Agent 无关（谁来都行，规矩在这）
= 真正模拟人类组织协作的 AI 工作区操作系统
```

## 附录：与 OpenClaw Gateway 的对比

### 共同点

1. **都自称"面向 AI Agent 的操作系统"**
   - OpenClaw Gateway："面向 AI Agent 的消息操作系统"
   - Agentic OS："面向 AI Agent 的工作区操作系统"
   - 两者都不满足于"调一次 API"的传统用法，都试图给 AI 一个常驻的、有状态的运行环境

2. **本地优先 + 文件系统持久化**
   - Gateway：所有 Session、配置、日志都在本地磁盘（`~/.openclaw/`），不经过任何云服务
   - Agentic OS：调用栈和 `.dna/` 都在项目文件系统里，不依赖任何平台
   - 两者都把"数据所有权归用户"作为基本原则

3. **都解决跨会话状态连续性**
   - Gateway：Session JSONL 记录 + Compaction + Memory Flush，Agent 可以回忆之前的对话
   - Agentic OS：`call-stack.json` + `.dna/wip.md`，AI 重启后可以恢复任务进度

4. **都有"Agent 身份"概念**
   - Gateway：每个 Agent 有 `SOUL.md`、`IDENTITY.md`，决定了 Agent 是谁
   - Agentic OS：每个模块有 `.dna/architecture.md`，决定了模块的职责和规则

5. **都强调与现有框架（LangChain/AutoGen/CrewAI）的差异**
   - 两者都明确把自己与"脚本式一次性调用"区分开来，强调有状态、可持续、有调度能力

### 核心区别

| 维度 | OpenClaw Gateway | Agentic OS |
|------|------------------|------------|
| **管的是什么** | 通道和消息——Agent 在哪里说话 | 工作区和上下文——Agent 看到什么、能改什么 |
| **核心抽象** | Agent Turn（一次完整的消息处理轮次） | Module + .dna/（工作内容单元及其身份编码） |
| **接入方式** | 多通道（WhatsApp/Telegram/飞书/Discord...） | MCP 协议（Cursor/Claude Code/Codex...） |
| **Agent 运行模型** | 单进程守护进程，7×24 常驻运行 | MCP Server，被 AI IDE 按需调用 |
| **状态单元** | Session（对话级） | Module（工作区级） |
| **安全模型** | 设备配对 + Token + TLS | 拓扑视界分级（物理级上下文隔离） |
| **调度机制** | 队列串行化（5 种队列模式）+ Cron | 调用栈（push/pop/suspend/resume） |
| **协作模型** | 同一 Gateway 托管多 Agent，Bindings 路由 | 模块 DAG 拓扑排序，依赖驱动执行 |
| **扩展方式** | 插件系统（TypeScript in-process） | .dna/ 规则文件（任何人/AI 可编辑） |
| **关注的 AI 问题** | 并发踩踏、消息重复、通道适配 | 上下文爆炸、边界失控、规则遗忘 |

### 设计哲学差异

**Gateway = 通信层操作系统**

Gateway 解决的核心问题是——Agent 怎么稳定地活在所有聊天通道里。它关注消息从哪来、往哪去（通道抽象），多条消息同时来怎么办（队列串行化），上一次聊到哪了（Session 管理），Agent 什么时候该主动干活（Cron 调度）。它的世界观是**以 Agent 为中心**的——Agent 是主角，通道/Session/工具都是为 Agent 服务的。

**Agentic OS = 认知层操作系统**

Agentic OS 解决的核心问题是——Agent 怎么在复杂工作区里不迷路、不越界、不遗忘。它关注工作区被切成了多少块、它们之间是什么关系（拓扑），当前 Agent 应该看到多少上下文（视界分级），手头的任务做到哪了（调用栈），这个工作区的规矩是什么（.dna/）。它的世界观是**以工作区为中心**的——工作区是主角，Agent 是可替换的执行者。

### 互补关系

两者解决的问题在不同层次，天然互补：

```
┌─────────────────────────────────────────────┐
│  用户 / 聊天通道 / 外部世界                    │
│  （WhatsApp、飞书、Discord...）               │
└──────────────────┬──────────────────────────┘
                   │
         ┌─────────▼──────────┐
         │  OpenClaw Gateway   │  ← 通信层：消息路由、Session、
         │  （Agent 运行时）    │     队列串行化、Cron 调度
         └─────────┬──────────┘
                   │  Agent 需要操作工作区时
         ┌─────────▼──────────┐
         │  Agentic OS         │  ← 认知层：上下文过滤、视界隔离、
         │  （MCP Server）     │     调用栈、.dna/ 规则
         └─────────┬──────────┘
                   │
         ┌─────────▼──────────┐
         │  工作区文件系统      │  ← 代码 / 美术 / 策划 / 音视频...
         └────────────────────┘
```

- Gateway 管 Agent 的"嘴"和"耳朵"——从哪听消息、往哪说话、什么时候主动行动
- Agentic OS 管 Agent 的"眼睛"和"大脑"——看到什么、记住什么、能碰什么、任务做到哪了

一个 Agent 如果同时接入两者：Gateway 让它 7×24 在线可达，Agentic OS 让它在工作区里有记忆、有约束、有层级感。

> **OpenClaw Gateway 让 AI 像一个"永远在线的团队成员"，Agentic OS 让 AI 像一个"有工作手册和权限管控的员工"。前者解决"怎么联系上 Agent"，后者解决"Agent 接手后怎么不搞砸"。**
